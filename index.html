  3. Import the "Cloud Requests" runtime module asset directly into your target project.

Optimized Transaction Timeline

The transaction workflow processes item distribution context entirely on the backend server. When a client checks out an item, run this simplified block sequence:

  1. Call transaction passing [Amount:Item] as Argument 1, and username as Argument 2.
  2. Verify the engine returns a SUCCESS token back to your local client script.
  3. Instantly dispatch a call to get_item passing username as Argument 1. The server handles verification context automatically.

Commands API

Endpoint Operational Context
get_captcha Requests an identity puzzle string to pass to the active project canvas.
captcha_sent Signals back-end workers to evaluate comment logs on the master project database.
get_owner Returns the registered owner account name connected to the active project context.
transaction Processes wallet balances and logs a pending receipt mapping to the Amount:Item parameters.
get_item Finalizes asset allocation, records it to the user data index, and clears the active transaction block.
get_inventory Fetches a comma-separated array string of verified items owned exclusively inside the calling project's system index.

Validation Rules

  • Signature Verification: Binds cloud execution threads directly to account credentials to prevent spoofing.
  • Scope Constraints: Asset queries run directly against independent node contexts to protect item ownership boundaries.
  • Deduplication Guard: Database limits enforce a strict singular-ownership system per asset name unless modified by custom server logic.

Troubleshooting Matrix

Exception State Correction Vector
Variables frozen on load Ensure project network credentials match an explicit entry in the backend config database.
Captcha validation drops Verify that client comment match fields are completely identical to the raw puzzle code sequence.
ERROR_NO_PENDING_TRANSACTION Ensure get_item is not invoked before transaction registers a confirmed SUCCESS state.
Handshake Dropouts Server load spikes can interrupt runtime channels. Reset project listeners via the green flag to clear variables.
