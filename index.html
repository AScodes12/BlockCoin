BlockCoin Documentation

Welcome to the official documentation for the BlockCoin currency system for Scratch. This system uses scratchattach to connect local instances and games to a centralized network cluster, providing synchronized cloud wallets and verified database exchanges safely.

How to Add BlockCoin to Your Project

  1. Visit the official BlockCoin network architecture center at scratch.mit.edu/projects/1331635541/.
  2. Open your Backpack dock at the bottom of the Scratch project layout.
  3. Drag the "Cloud Requests" asset module down into your backpack, enter your game platform, and unpack the sprite inside.
  4. Required Credit: You must explicitly credit @Timmccool inside your project deployment notes for providing the underlying communication engine framework.

How to Get Approval to Use BlockCoin

To preserve cross-game data sync integrity, all cloud pipelines must be explicitly whitelisted by our back-end listener before handling live calls.

  1. Submit an application on the official BlockCoin communications thread.
  2. The authorization process and database logging routing configuration typically take 24 to 48 hours to complete.
NOTICE: The centralized engine will drop variable responses originating from unauthorized project contexts automatically.

How to Use BlockCoin & Asset Integration

To run data mutations or inventory updates using the network, implement the PayCoins (Amount) (Item) custom block block stack.

This block triggers an internal identification query to our backend. Users will receive a generated authentication comment captcha code output inside this custom call workflow, which must be updated on the central project verification thread to execute asset clearance.

SHOP DEVELOPERS: OPTIMIZED TRANSACTION TIMELINE
The transaction workflow processes item distribution context entirely on the backend server. When a client checks out an item, run this simplified block sequence:
  1. Call transaction passing [Amount:Item] as Argument 1, and username as Argument 2.
  2. Verify the engine returns a SUCCESS token back to your local client script.
  3. Instantly dispatch a call to get_item passing username as Argument 1. The server handles verification context automatically.

BlockCoin Commands API

These are the core endpoints managed by the system parser inside the Cloud Requests module wrapper:

Command Name What It Does
get_captcha Requests an authentication puzzle token and returns it to your game project client.
captcha_sent Signals the back-end worker thread to verify comment strings on the master ledger project.
get_owner Looks up the registered host username of the calling project to balance credit allocations.
transaction Validates balances and creates a temporary purchase receipt using the Amount:Item structure.
get_item Finalizes the transaction workflow by claiming the user's pending asset and adding it directly to their long-term database file.
get_inventory Returns a comma-separated string containing every validated item owned by the user exclusively inside the calling project's environment.

System Consistency & Purchase Rules

BlockCoin implements architecture layers to maintain stable ledger profiles across all instances:

  • Identity Syncing: Securely binds actions to platform data signatures to prevent impersonation.
  • State Validation: Compares pending purchase parameters against real-time node IDs to stop spoofing tricks.
  • Purchase Limits: Database checks ensure specific items can only be bought once per user per project. Developers should parse get_inventory to verify ownership states before offering checkout menus.

Quick Troubleshooting

Issue Encountered Resolution Path
Transactions are completely stuck on loading Verify that your project ID has been registered in our central config network file. Unapproved IDs are automatically blocked.
Verification codes are failing Ensure the player posted the precise alphanumeric string directly as a main comment on the master profile project.
Items return ERROR_NO_PENDING_TRANSACTION Ensure your script does not call get_item unless transaction has completed with a SUCCESS response code. Check that the player does not already own the item, as duplicates are rejected.
Cloud variable connection dropped Network drops can happen during Scratch traffic peaks. The backend queue handles variations, but users can safely click green flag triggers to restart communication handshakes.
